# Skill 优化方法论

## 优化触发信号

- 用户说"这个 skill 不好用"
- 测评发现的问题需要修复
- 用户想增加新功能或改进行为
- Agent 频繁犯同一类错

## 优化流程

### 第一步：诊断

先用「测评」流程找出问题，按优先级排列：

1. **安全问题** — 最优先
2. **触发问题** — 该触发没触发 / 不该触发却触发了
3. **流程问题** — 步骤遗漏、顺序错误
4. **输出问题** — 格式错、内容缺
5. **效率问题** — Token 浪费、响应慢

### 第二步：制定方案

每个问题对应具体改动：

| 问题类型 | 优化方向 | 改动位置 |
|----------|----------|----------|
| 触发不准 | 改触发词，补正例/反例 | description + Gotchas |
| 步骤遗漏 | 补步骤，加检查点 | workflow 章节 |
| 输出格式差 | 改格式，补模板 | 输出规范 + 示例 |
| Token 浪费 | 精简正文，长资料放 references/ | 正文结构 |
| 误触发 | 收窄触发条件 | 触发条件 + 不触发条件 |
| 安全风险 | 移除敏感引用，限制权限 | allowed-tools + 正文 |
| 缺错误处理 | 补失败处理 | 失败处理表格 |

### 第三步：执行改进

**改 description 的注意事项：**
- 改完必须同步补正例和反例
- 验证路由是否真的改善
- 前 250 字符仍然是重点

**改 workflow 的注意事项：**
- 每步必须有明确动作
- 加检查点而非加原则
- 正文仍 ≤ 500 行

**精简 Token 的方法：**
- 长示例移到 `examples/`
- 详细规范移到 `references/`
- 正文只保留必要流程
- 用 `${CLAUDE_SKILL_DIR}` 引用

### 第四步：验证

1. **同一个 case 跑一遍** — 确认问题解决
2. **换一种说法触发** — 确认触发条件没变窄太多
3. **用 evals 回归** — 确认没引入新问题

### 第五步：记录

在 Skill 中添加 Gotchas，记录优化过程中发现的坑：

```markdown
## Gotchas

### G1: 看起来像但不该触发
- 用户说"xxx"可能是问功能，不是要优化
- 判断依据：是否提到"改进"、"不好用"等关键词

### G2: 容易误改的地方
- 改 description 后必须同步改 evals
- 不要只改触发词不改示例
```

## 泛化与特化时机

- **从特化开始** — 先覆盖一个具体场景
- **泛化条件** — 同一规则在 3+ 个不同 case 里重复出现
- **特化信号** — 泛化后误触发率上升

## 什么时候该退役

- 模型原生能力已覆盖该场景
- 维护成本超过收益
- 流程已经过时
- 和新 Skill 大量重叠

**退役不是失败。** Skill 太多互相打架，Agent 反而更难做对。
