# Skill 优化方法论

## 核心理念：棘轮机制

**分数只能上升，每一轮要么改进 Skill，要么干净地回滚。**

棘轮机制是 Skill 优化迭代的核心保障——就像机械棘轮一样，只允许朝一个方向转动。如果某轮优化未能带来可量化的提升，系统将自动回滚到上一个已验证状态，保证交付质量单调递增。

```
┌─────────────────────────────────────────────────────────┐
│                   棘轮优化循环                            │
│                                                         │
│   ┌───────────┐    ┌───────────┐    ┌───────────┐      │
│   │ Phase 0   │───→│ Phase 1   │───→│ Phase 2   │      │
│   │ 诊断评估   │    │ 方案制定   │    │ 执行改进   │      │
│   └───────────┘    └───────────┘    └───────────┘      │
│         ↑                                  │            │
│         │                                  ↓            │
│   ┌───────────┐                   ┌───────────┐        │
│   │ Phase 3   │←──────────────────│ 多评委审查  │        │
│   │ 决策与记录 │                   └───────────┘        │
│   └───────────┘                                         │
│         │                                                │
│         ↓                                                │
│   [通过] → 进入下一轮  |  [退步] → 回滚到上一版本         │
│   [触顶] → 停止迭代   |  [违规] → 反例黑名单拦截         │
└─────────────────────────────────────────────────────────┘
```

---

## 优化触发信号

- 用户说"这个 skill 不好用"
- 测评发现的问题需要修复
- 用户想增加新功能或改进行为
- Agent 频繁犯同一类错
- 评分触顶后用户主动要求继续优化

---

## 优化流程（Phase 0 – Phase 3）

### Phase 0：诊断与评估

先用「测评」流程找出问题，按优先级排列：

1. **安全问题** — 最优先
2. **触发问题** — 该触发没触发 / 不该触发却触发了
3. **流程问题** — 步骤遗漏、顺序错误
4. **输出问题** — 格式错、内容缺
5. **效率问题** — Token 浪费、响应慢

**具体操作：**

- 分析当前 SKILL.md 的结构完整性
- 检查各章节是否清晰、完整、无歧义
- 识别可量化的问题点（缺失章节、模糊描述、冗余内容等）
- 运行 test-prompts 获取实际输出，基于输出评分
- 输出：诊断报告 + 上一轮基线分数（首轮无基线则为 0）

**禁止事项：**

- 不得跳过 test-prompts 直接评分（AP-04）
- 不得在同 context 中自评自改（AP-01）

---

### Phase 1：方案制定

每个问题对应具体改动：

| 问题类型 | 优化方向 | 改动位置 |
|----------|----------|----------|
| 触发不准 | 改触发词，补正例/反例 | description + Gotchas |
| 步骤遗漏 | 补步骤，加检查点 | workflow 章节 |
| 输出格式差 | 改格式，补模板 | 输出规范 + 示例 |
| Token 浪费 | 精简正文，长资料放 references/ | 正文结构 |
| 误触发 | 收窄触发条件 | 触发条件 + 不触发条件 |
| 安全风险 | 移除敏感引用，限制权限 | allowed-tools + 正文 |
| 缺错误处理 | 补失败处理 | 失败处理表格 |

**关键约束：**

- 每轮只改一个维度（AP-05），不得同时改触发词 + 流程 + 输出格式
- 优先处理高优先级问题
- 改进应具有原子性——每次改一个明确的点
- 优化后 SKILL.md 大小 ≤ 原始大小 × 1.5（防膨胀约束）
- 输出：改进计划（变更 diff）

---

### Phase 2：执行改进

将改进计划应用到 SKILL.md，保留变更前的快照用于回滚。

**改 description 的注意事项：**
- 改完必须同步补正例和反例
- 验证路由是否真的改善
- 前 250 字符仍然是重点

**改 workflow 的注意事项：**
- 每步必须有明确动作
- 加检查点而非加原则
- 正文仍 ≤ 500 行

**精简 Token 的方法：**
- 长示例移到 `examples/`
- 详细规范移到 `references/`
- 正文只保留必要流程
- 用 `${SKILL_DIR}` 引用

**Phase 2 改动前反例检查（必做）：**

```
□ AP-01: 是否在新 context 中评分？（不得同 context 自评自改）
□ AP-02: 回滚方式是否安全？（不用 reset --hard，用 stash 或 checkout）
□ AP-03: 改动是否为真实质量提升？（不为凑分增冗余）
□ AP-04: 是否已运行 test-prompts？（不得跳过测试直接评分）
□ AP-05: 本轮是否只改一个维度？（不得轮内改多个维度）
□ AP-06: dry_run 比例是否 ≤ 30%？（主要靠实际执行）
□ AP-07: 异常是否已记录和处理？（不得静默跳过异常）
□ AP-08: 是否考虑了维度相关性？（不得孤立优化）

全部通过 → 继续改动
任一未通过 → 先解决问题
```

**回滚方式（AP-02）：**
```bash
# 回滚单个文件
git checkout -- SKILL.md
# 或暂存所有改动
git stash
# 或创建临时分支保存
git checkout -b backup-branch
git checkout main
```

### Phase 2.5：多评委独立审查

> **核心依据：LLM 自评准确率仅 46.4%（SkillLens 论文实证），必须引入多评委交叉验证。**

Phase 2 执行完成后，进入多评委独立审查。

**评委规则：**
- 每轮启动 **至少 2 个独立评委**，互不干扰
- 评委不可复用，下一轮必须 spawn 新评委
- 至少 2 个评委共识才有效

**执行流程：**

```
主 Agent
    │
    ├── spawn 子 Agent A（评委 A） → 独立评估 → 评分 + 理由
    └── spawn 子 Agent B（评委 B） → 独立评估 → 评分 + 理由
```

**评委提示模板：**

```markdown
你是一个独立的 Skill 质量评委。请评估以下 Skill 的质量，输出：
1. 评分（1-10 分）
2. 评分理由（必须基于具体问题）
3. 发现的问题列表

评估维度：
- 触发准确性
- 流程完整性
- 输出质量
- 安全性
- Token 效率

请独立评估，不要参考任何其他评分结果。
```

**共识机制：**

| 分差 | 处理方式 | 说明 |
|------|----------|------|
| ≤ 1 | 取平均分 | 评委高度一致 |
| = 2 | 取中位数 | 有一定分歧 |
| > 2 | 启动第 3 个评委仲裁 | 分歧过大，需要仲裁 |

**干跑模式（子 Agent 不可用时）：**

```
干跑模式评估
├── 模拟评委 A 的视角（基于规则预设的评分标准）
├── 模拟评委 B 的视角（基于不同规则预设的评分标准）
└── 标注: [DRY_RUN] 结果仅供参考，建议子 agent 可用后重新评估
```

干跑比例控制：
- ≤ 30%：正常继续
- 30%-50%：告警 + 人工复核建议
- \> 50%：暂停评估 + 强制人工介入

### Phase 3：决策与记录

#### 棘轮决策

| 条件 | 决策 | 动作 |
|------|------|------|
| 本轮分 > 上轮分 | ✅ 通过 | 保留本轮版本，进入下一轮优化 |
| 本轮分 = 上轮分 | ❌ 回滚 | 回滚到上一版本，本轮改动丢弃 |
| 本轮分 < 上轮分 | ❌ 回滚 | 回滚到上一版本，本轮改动丢弃 |

**文件大小约束：**
```
优化后大小 ≤ 原始大小 × 1.5
```
若超过 150% 限制，该轮视为"不通过"，需在下一轮精简。精简优先级：
1. 冗余描述
2. 过于详细的示例（保留关键示例即可）
3. 重复说明同一概念的内容

#### 触顶信号检测

**当优化收益递减至临界点时，自动停止迭代。**

```
连续 2 轮 Δ < 2 分 → 自动停止优化
```

- **Δ** = 本轮分数 - 上轮分数
- 连续 2 轮增量 Δ 均小于 2 分，判定为"已触顶"
- 触顶后不再进行新一轮优化，当前版本即为最终版本
- 避免无效迭代：分数提升极小时，继续优化的投入产出比过低
- 保护已有成果：接近最优时，激进改动反而可能引入退步

#### 记录

在 Skill 中添加 Gotchas，记录优化过程中发现的坑：

```markdown
## Gotchas

### G1: 看起来像但不该触发
- 用户说"xxx"可能是问功能，不是要优化
- 判断依据：是否提到"改进"、"不好用"等关键词

### G2: 容易误改的地方
- 改 description 后必须同步改 evals
- 不要只改触发词不改示例
```

每轮优化保留完整的变更记录：
- 变更前的 SKILL.md 快照
- 本轮的诊断结果和改进方案
- 变更后的 SKILL.md 快照
- 多评委评估报告
- 棘轮决策（通过/回滚）

形成完整的 `v1 → v2 → v3 → ...` 版本链。

## 泛化与特化时机

- **从特化开始** — 先覆盖一个具体场景
- **泛化条件** — 同一规则在 3+ 个不同 case 里重复出现
- **特化信号** — 泛化后误触发率上升

## 什么时候该退役

- 模型原生能力已覆盖该场景
- 维护成本超过收益
- 流程已经过时
- 和新 Skill 大量重叠

**退役不是失败。** Skill 太多互相打架，Agent 反而更难做对。
