# Darwin 评分标准 (Darwin Rubric)

## 概述

本评分标准基于微软 SkillLens (arXiv:2605.23899) 论文的实证验证，该研究系统性地分析了模型生成的 Agent 技能的全生命周期——从经验生成、技能提取到技能消费。论文发现：

- **模型生成的技能平均有益，但存在非平凡的负面迁移**
- **技能效用独立于模型规模或基线任务强度**
- **具体补救措施比通用建议更有效**
- **Meta-Skill 指导可显著提升技能质量并减少负面迁移**

本评分标准将这些实证发现转化为可操作的 9 维度评估框架，用于评估 Skill 的结构质量和实际效果。

---

## 评分维度

### 结构维度（维度 1-7）— 共 70 分

#### 维度 1：结构完整性（Structural Completeness）— 10 分

**理论依据**：SkillLens 研究发现，技能的结构化程度直接影响其可复用性和跨模型迁移效果。

**检查项**：
- [ ] 遵循 SKILL.md 标准格式（YAML frontmatter + Markdown 正文）
- [ ] 包含必要的元数据字段（name、description、version）
- [ ] 目录结构完整（references/、scripts/、evals/ 等）
- [ ] 正文 ≤500 行，避免过度冗长
- [ ] 使用 ${SKILL_DIR} 引用文件

**评分标准**：
| 分数段 | 标准 |
|--------|------|
| 9-10 | 结构完整，符合官方规范，模块化清晰 |
| 7-8 | 结构基本完整，有小缺漏 |
| 4-6 | 结构不完整，缺少关键组件 |
| 0-3 | 无结构，内容堆砌 |

#### 维度 2：指令具体性（Instruction Specificity）— 10 分

**理论依据**：SkillLens 论文 §5.2 发现，"具体补救措施，而非通用建议"是有效技能的关键特征。文本表面质量不能预测技能效用，具体性才是关键。

**检查项**：
- [ ] 指令具体、可执行，而非泛泛而谈
- [ ] 提供具体的步骤、命令或代码示例
- [ ] 避免"应该"、"建议"等模糊词汇，使用"必须"、"执行"等明确指令
- [ ] 包含具体的输入/输出示例
- [ ] 针对特定场景而非通用情况

**评分标准**：
| 分数段 | 标准 |
|--------|------|
| 9-10 | 指令高度具体，可直接执行，无歧义 |
| 7-8 | 指令基本具体，偶有模糊 |
| 4-6 | 指令过于通用，缺乏可执行性 |
| 0-3 | 指令模糊，无法执行 |

#### 维度 3：失败模式编码（Failure Mode Encoding）— 10 分

**理论依据**：SkillLens §5.1 发现，失败轨迹与成功轨迹同样重要，甚至更能提升技能质量。有效的技能应编码常见的失败模式和避坑指南。

**检查项**：
- [ ] 包含"不要做"的明确警告
- [ ] 列出常见错误和反模式
- [ ] 提供错误识别和修复方法
- [ ] 包含边界情况的处理策略
- [ ] 有失败案例的示例

**评分标准**：
| 分数段 | 标准 |
|--------|------|
| 9-10 | 完整编码失败模式，提供具体避坑指南 |
| 7-8 | 包含主要失败模式，但不够全面 |
| 4-6 | 失败模式编码不足，缺少关键警告 |
| 0-3 | 无失败模式编码 |

#### 维度 4：可执行具体性（Executable Specificity）— 10 分

**理论依据**：SkillLens 研究表明，技能的实用性取决于其可执行程度。抽象的知识不如具体的执行步骤有效。

**检查项**：
- [ ] 提供可直接复制执行的命令或代码
- [ ] 包含完整的执行上下文（环境、依赖、前置条件）
- [ ] 步骤编号清晰，顺序明确
- [ ] 包含验证步骤，确认执行成功
- [ ] 提供回滚或修复方案

**评分标准**：
| 分数段 | 标准 |
|--------|------|
| 9-10 | 高度可执行，可直接复制运行 |
| 7-8 | 基本可执行，需少量调整 |
| 4-6 | 可执行性一般，需要较多补充 |
| 0-3 | 无法执行，过于抽象 |

#### 维度 5：边界条件（Boundary Conditions）— 10 分

**理论依据**：SkillLens §5.3 发现，技能的跨模型迁移效果差异显著。明确的边界条件有助于评估技能的适用范围。

**检查项**：
- [ ] 明确说明适用场景和不适用场景
- [ ] 列出前置条件和依赖要求
- [ ] 定义输入/输出的格式和约束
- [ ] 说明性能限制和资源要求
- [ ] 标注版本兼容性

**评分标准**：
| 分数段 | 标准 |
|--------|------|
| 9-10 | 边界条件清晰，适用范围明确 |
| 7-8 | 边界条件基本清晰，偶有模糊 |
| 4-6 | 边界条件不明确，容易误用 |
| 0-3 | 无边界条件说明 |

#### 维度 6：用户确认点（User Confirmation Points）— 10 分

**理论依据**：SkillLens 论文强调，技能消费阶段需要目标模型的积极参与。明确的确认点有助于确保技能被正确应用。

**检查项**：
- [ ] 在关键操作前设置确认点
- [ ] 提供决策树或条件分支
- [ ] 允许用户自定义参数
- [ ] 包含"如果...则..."的条件逻辑
- [ ] 提供跳过或自定义选项

**评分标准**：
| 分数段 | 标准 |
|--------|------|
| 9-10 | 确认点设计合理，用户体验流畅 |
| 7-8 | 有基本确认点，但不够完善 |
| 4-6 | 确认点不足，容易误操作 |
| 0-3 | 无确认点，完全自动化 |

#### 维度 7：高风险行动黑名单（High-Risk Action Blacklist）— 10 分

**理论依据**：SkillLens §4.2 发现，技能可能产生负面迁移。高风险行动黑名单是防止负面迁移的关键安全机制。

**检查项**：
- [ ] 列出禁止执行的危险操作
- [ ] 提供风险评估矩阵
- [ ] 包含安全检查清单
- [ ] 定义回滚策略
- [ ] 标注不可逆操作

**评分标准**：
| 分数段 | 标准 |
|--------|------|
| 9-10 | 黑名单完整，风险控制严格 |
| 7-8 | 有基本黑名单，但不够全面 |
| 4-6 | 黑名单不足，存在安全隐患 |
| 0-3 | 无黑名单，高风险操作无限制 |

---

### 效果维度（维度 8）— 共 20 分

#### 维度 8：实测表现（Empirical Performance）— 20 分

**理论依据**：SkillLens 的核心贡献是提出 Extraction Efficacy (EE) 和 Target Evolvability (TE) 两个指标，通过实测验证技能的实际效果。

**检查项**：
- [ ] 有 evals 评测集（至少 5 条：2 core + 1 boundary + 2 edge）
- [ ] 提供成功率或性能指标
- [ ] 包含基线对比数据
- [ ] 有跨模型/跨场景测试结果
- [ ] 记录负面迁移案例

**评分标准**：
| 分数段 | 标准 |
|--------|------|
| 18-20 | 有完整评测集，实测数据充分，效果显著 |
| 14-17 | 有基本评测，效果良好 |
| 10-13 | 评测不足，效果一般 |
| 0-9 | 无评测，效果未知 |

---

### Meta-Skill 维度（维度 9）— 共 10 分

#### 维度 9：反例黑名单检查（Anti-Example Blacklist Check）— 10 分

**理论依据**：SkillLens §6 提出的 Meta-Skill 指导提取方法，通过反例识别和黑名单机制，显著减少负面迁移并提升技能质量。

**检查项**：
- [ ] 包含明确的反例（不该触发的情况）
- [ ] 列出常见的误用场景
- [ ] 提供反例的识别方法
- [ ] 有反例的处理策略
- [ ] 定期更新反例库

**评分标准**：
| 分数段 | 标准 |
|--------|------|
| 9-10 | 反例库完整，覆盖主要误用场景 |
| 7-8 | 有基本反例，但不够全面 |
| 4-6 | 反例不足，容易误用 |
| 0-3 | 无反例说明 |

## 总分计算

### 加权总分公式

```
总分 = (维度1 × 1.0) + (维度2 × 1.0) + (维度3 × 1.0) + (维度4 × 1.0) + 
       (维度5 × 1.0) + (维度6 × 1.0) + (维度7 × 1.0) + (维度8 × 2.0) + (维度9 × 1.0)
```

**满分**：100 分

### 维度权重说明

| 维度 | 权重 | 说明 |
|------|------|------|
| 维度 1-7（结构维度） | 各 1.0 | 基础结构质量，共 70 分 |
| 维度 8（效果维度） | 2.0 | 实际效果是核心，共 20 分 |
| 维度 9（Meta-Skill 维度） | 1.0 | 元能力保障，共 10 分 |

## 使用方式

### 1. 评估流程

```
安全审查 → 结构维度评分 → 效果维度评分 → Meta-Skill 维度评分 → 生成报告
```

**安全审查是前置条件**：发现安全问题（BLOCK）则停止，不进入质量评分。

### 2. 评分步骤

#### 步骤 1：安全审查
- 检查是否有敏感信息泄露
- 检查是否有命令注入风险
- 检查是否有危险操作
- 确定安全评级（SAFE/WARNING/DANGER/BLOCK）

#### 步骤 2：结构维度评分（维度 1-7）
- 逐项检查每个维度的检查项
- 根据评分标准给出分数
- 记录扣分原因和改进建议

#### 步骤 3：效果维度评分（维度 8）
- 检查 evals 评测集
- 评估实测数据
- 验证效果指标

#### 步骤 4：Meta-Skill 维度评分（维度 9）
- 检查反例库
- 评估误用防护
- 验证黑名单完整性

#### 步骤 5：生成报告
- 计算加权总分
- 确定综合评级
- 输出评分报告

### 3. 报告模板

```markdown
# Darwin 评分报告 — {skill名称}

**评估时间：** {日期}
**Skill 路径：** {路径}
**Skill 版本：** {版本或"未标注"}
```

### 4. 评分注意事项

#### 必须遵守
- **客观公正**：基于事实和规范，不带个人偏好
- **具体明确**：指出具体行号和代码
- **建设性**：提供改进建议，不只是批评
- **对照规范**：每条建议必须引用官方规范或知识库条目

#### 禁止行为
- 人身攻击
- 模糊的批评（如"这个 skill 不好"）
- 忽略上下文（不考虑 skill 的设计意图）
- 无依据打分（每个分数必须有对应检查项支撑）

### 5. 与其他评分标准的关系

本评分标准与 `skill-review` 的 10 维度评分标准互补：

| 评分标准 | 侧重点 | 适用场景 |
|----------|--------|----------|
| Darwin Rubric | 结构质量和实证效果 | 评估技能的工程质量和实际效果 |
| skill-review 10 维度 | 规范性和实用性 | 评估技能的规范符合度和实用性 |

**建议**：对于重要技能，可同时使用两种评分标准进行交叉验证。
