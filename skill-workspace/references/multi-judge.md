# 多评委评估机制（Multi-Judge）

## 概述

基于 SkillLens 论文实证研究，**LLM 自评准确率仅 46.4%**。这意味着让模型自己评估自己的输出，几乎和抛硬币一样不可靠。

为解决这一问题，采用**多评委独立评估**机制：每轮评估启动多个独立评委 agent，通过交叉验证和共识机制确保评估结果的可靠性和一致性。

## 核心规则

### R1: 每轮启动 2 个独立评委

每轮评估必须启动 **至少 2 个独立评委**，每个评委独立打分，互不干扰。

**目的：** 避免单点偏差，通过多视角评估提高准确率。

### R2: 评委不复用

同一轮评估中，**评委不能复用**。每个评委只能使用一次，下一轮评估必须重新 spawn 新的评委。

**目的：** 防止评委被污染（记忆前一轮的打分结果），确保每轮评估的独立性。

### R3: 至少 2 个 judge 共识才信

评估结果必须获得 **至少 2 个评委的共识** 才算有效。如果 2 个评委打分差异过大（如差距 > 2 分），需要启动第 3 个评委进行仲裁。

**共识阈值：**
- 分差 ≤ 1：取平均分
- 分差 = 2：取中位数
- 分差 > 2：启动第 3 个评委仲裁

## 实现流程

### 第一步：评分时 spawn 子 agent

在需要评估 Skill 质量时，主 agent 通过子 agent 机制启动独立评委：

```
主 Agent（skill-improver / skill-reviewer）
    │
    ├── spawn 子 Agent A（评委 A）
    │   └── 独立评估 → 输出评分 + 理由
    │
    └── spawn 子 Agent B（评委 B）
        └── 独立评估 → 输出评分 + 理由
```

**子 Agent 启动参数：**

| 参数 | 值 | 说明 |
|------|-----|------|
| isolation | true | 确保评委之间互不干扰 |
| context | minimal | 只传入待评估内容，不传历史评分 |
| output | score + reason | 必须输出评分和理由 |

**子 Agent 提示模板：**

```markdown
你是一个独立的 Skill 质量评委。请评估以下 Skill 的质量，输出：
1. 评分（1-10 分）
2. 评分理由（必须基于具体问题）
3. 发现的问题列表

评估维度：
- 触发准确性
- 流程完整性
- 输出质量
- 安全性
- Token 效率

请独立评估，不要参考任何其他评分结果。
```

### 第二步：汇总分数

收集所有评委的评分结果，进行汇总：

```
评分汇总
├── 评委 A: 7.5 分（理由：流程完整，但输出格式不够规范）
├── 评委 B: 7.0 分（理由：触发词覆盖不足，流程基本完整）
│
├── 分差: 0.5（≤ 1，取平均分）
└── 最终得分: 7.25
```

**汇总规则：**

| 分差 | 处理方式 | 说明 |
|------|----------|------|
| ≤ 1 | 取平均分 | 评委高度一致 |
| = 2 | 取中位数 | 有一定分歧 |
| > 2 | 启动第 3 个评委 | 分歧过大，需要仲裁 |

### 第三步：下一轮重新 spawn

完成一轮评估后，**必须销毁当前评委**，下一轮评估时重新 spawn 新的评委：

```
第一轮评估
├── spawn 评委 A → 评分 → 销毁
├── spawn 评委 B → 评分 → 销毁
└── 汇总结果

第二轮评估（Skill 修改后）
├── spawn 评委 C → 评分 → 销毁
├── spawn 评委 D → 评分 → 销毁
└── 汇总结果

... 重复直到满足退出条件
```

**退出条件：**
- 连续 2 轮评分 ≥ 8 分
- 达到最大迭代轮次（默认 5 轮）
- 用户手动终止

## 干跑模式

### 触发条件

当子 agent 不可用时（如环境限制、权限不足），自动切换到**干跑模式**。

**检测方法：**

```bash
# 检查子 agent 是否可用
if ! command -v agent &> /dev/null; then
    echo "子 agent 不可用，切换到干跑模式"
    DRY_RUN=true
fi
```

### 干跑验证流程

在干跑模式下，主 agent **模拟评委行为**，但会标注这是干跑结果：

```
干跑模式评估
├── 模拟评委 A 的视角
│   └── 基于规则预设的评分标准
├── 模拟评委 B 的视角
│   └── 基于不同规则预设的评分标准
│
└── 标注: [DRY_RUN] 结果仅供参考，建议子 agent 可用后重新评估
```

**干跑模式输出格式：**

```markdown
## 评估结果 [DRY_RUN]

**评委 A（模拟）:** 7.0 分
- 理由：基于规则 R1、R3 的评估结果
- 发现问题：X、Y、Z

**评委 B（模拟）:** 6.5 分
- 理由：基于规则 R2、R4 的评估结果
- 发现问题：X、Y

**汇总:** 6.75 分（干跑模式，仅供参考）

⚠️ 注意：这是干跑模式的结果，建议在子 agent 可用后重新评估。
```

## 干跑比例控制

### 告警阈值

**当干跑比例 > 30% 时，自动触发告警。**

**计算公式：**

```
干跑比例 = 干跑评估次数 / 总评估次数 × 100%
```

### 告警机制

```
干跑比例监控
├── 实时统计干跑次数和总次数
├── 每次评估后计算当前比例
└── 比例 > 30% → 触发告警
    ├── 输出告警信息
    ├── 建议检查子 agent 环境
    └── 标记评估结果可信度降低
```

**告警输出格式：**

```markdown
⚠️ 干跑比例告警

当前干跑比例: 35% (7/20)
阈值: 30%

问题: 子 agent 长时间不可用，评估结果可信度下降。

建议:
1. 检查子 agent 环境是否正常
2. 确认是否有权限限制
3. 考虑手动评估关键 Skill

已标记: 后续评估结果需人工复核
```

### 告警处理策略

| 干跑比例 | 处理策略 | 说明 |
|----------|----------|------|
| ≤ 30% | 正常继续 | 可信度可接受 |
| 30%-50% | 告警 + 人工复核建议 | 可信度下降 |
| > 50% | 暂停评估 + 强制人工介入 | 可信度过低 |

## 最佳实践

### 实践 1: 评委多样性

确保评委使用不同的评估视角和规则：

```
评委 A: 侧重流程完整性
评委 B: 侧重输出质量
评委 C（仲裁）: 侧重整体平衡
```

### 实践 2: 评估结果缓存

为提高效率，可以缓存最近一次有效评估结果：

```
缓存策略
├── 缓存有效期: Skill 未修改时
├── 缓存失效: Skill 文件变更时
└── 强制刷新: 用户主动请求时
```

### 实践 3: 评估历史记录

记录每轮评估的详细信息，便于追溯和分析：

```json
{
  "skill_name": "skill-reviewer",
  "round": 1,
  "timestamp": "2026-06-13T10:30:00Z",
  "judges": [
    {"id": "judge_a", "score": 7.5, "dry_run": false},
    {"id": "judge_b", "score": 7.0, "dry_run": false}
  ],
  "final_score": 7.25,
  "consensus": true,
  "issues_found": ["trigger_coverage", "output_format"]
}
```
