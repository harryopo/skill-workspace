# 需求深挖指南

## 概述

需求深挖是 Skill 开发的第一步，目标是深入理解用户的真实痛点和需求，而不是只问几个表面问题。

## "无需求不开发"强制检查

### 核心原则

**在开始任何开发前，必须先确认用户需求。**

### 检查流程

1. **需求接收**
   - 用户提出需求时，先判断是否清晰完整
   - 如果用户说的很少，主动提问引导

2. **需求澄清**
   - 必须获得用户确认后才进入下一阶段
   - 不能假设用户想要什么，必须明确询问

3. **确认标准**
   - 功能范围明确
   - 技术方案确认
   - 优先级确定

### 触发条件

- 用户说"做个 xxx"
- 用户说"帮我实现 xxx"
- 用户说"开发一个 xxx"
- 任何开发任务开始前

## 合并 Skill 的需求澄清流程

### 触发条件

当用户说以下类似内容时触发：
- "合并 xxx 和 yyy skill"
- "把 xxx 和 yyy 合在一起"
- "整合 xxx 和 yyy"

### 澄清流程

#### 第一步：审查原 Skill

调用 review 子技能，审查被合并的 skill：
- 检查每个 skill 的质量
- 识别优点和问题
- 评估合并可行性

#### 第二步：理解原 Skill

阅读每个 skill 的 SKILL.md 文件：
- 理解功能定位
- 掌握使用场景
- 了解技术实现

#### 第三步：询问用户

向用户确认以下问题：

1. **保留优点**
   - 两个 skill 各自有哪些优点需要保留？
   - 有没有特别喜欢的功能或设计？

2. **解决痛点**
   - 使用中遇到过什么问题？
   - 有哪些地方不满意？

3. **新需求**
   - 合并后希望增加什么新功能？
   - 有没有新的使用场景？

4. **优先级**
   - 哪些功能是必须的？
   - 哪些可以妥协？

#### 第四步：确认后继续

- 整理用户反馈
- 确认合并方案
- 获得用户明确同意后进入开发阶段

### 输出格式

```markdown
# 合并需求文档

## 原 Skill 审查

### Skill A: [名称]
- 优点：[优点列表]
- 问题：[问题列表]

### Skill B: [名称]
- 优点：[优点列表]
- 问题：[问题列表]

## 用户需求

### 保留优点
- [用户要求保留的功能]

### 解决痛点
- [用户希望改进的地方]

### 新增需求
- [用户希望添加的功能]

## 合并方案

### 功能整合
- [整合后的功能列表]

### 技术方案
- [技术实现方案]

## 确认结果

用户确认：[是/否]
确认时间：[时间]
```

## 痛点挖掘（5-8 个问题）

### 问题清单

1. **这个任务你重复做过几次？**
   - 选项：1次 / 2-5次 / 6-10次 / 10次以上
   - 目的：判断是否值得 Skill 化（至少 3 次才值得）

2. **每次做的时候最痛苦的是什么？**
   - 开放式问题
   - 目的：找到核心痛点

3. **你理想中的解决方案是什么样的？**
   - 开放式问题
   - 目的：理解用户期望

4. **有没有类似的工具？哪里不满意？**
   - 开放式问题
   - 目的：了解竞品，找到差异化

5. **你的使用场景是什么？**
   - 选项：个人 / 团队 / 企业
   - 目的：确定功能范围

6. **有没有特殊要求？**
   - 开放式问题
   - 目的：识别约束条件

7. **你希望达到什么效果？**
   - 开放式问题
   - 目的：明确成功标准

8. **有没有时间/预算限制？**
   - 开放式问题
   - 目的：评估可行性

### 提问策略

- **一次只问一个问题** - 不要一次问多个问题，让用户专注于一个问题
- **优先使用多选题** - 比开放式问题更容易回答
- **根据回答调整后续问题** - 如果用户已经回答了某个问题，跳过
- **追问细节** - 如果回答模糊，追问具体细节

### 智能化提问策略

#### 用户说的少时的应对

当用户回复简短（如"好"、"可以"、"随便"）时：

1. **利用 LLM 能力主动提问**
   - 分析用户意图，推测可能的需求
   - 提供具体选项供用户选择
   - 用示例帮助用户理解

2. **根据用户回答调整后续问题**
   - 如果用户选择简单方案，后续问题更简洁
   - 如果用户选择复杂方案，深入询问细节
   - 根据用户技术水平调整问题难度

3. **追问细节，确保理解准确**
   - 对模糊回答进行具体化追问
   - 用"您是指...还是...？"的方式澄清
   - 确认理解后复述用户需求

#### 智能提问示例

**场景1：用户说"做个工具"**
```
用户：做个工具
助手：好的，我来帮您梳理需求。
您希望这个工具解决什么问题？
A. 提高工作效率
B. 自动化重复任务
C. 数据处理分析
D. 其他（请描述）
```

**场景2：用户说"优化一下"**
```
用户：优化一下
助手：明白，您想优化哪个方面？
A. 性能（速度、内存）
B. 用户体验（操作流程）
C. 代码质量（可维护性）
D. 功能完善（缺失功能）
```

**场景3：用户回复模糊**
```
用户：好一点就行
助手：理解，您说的"好一点"具体是指：
A. 运行更快
B. 界面更美观
C. 功能更完整
D. 使用更简单
```

#### 提问节奏控制

1. **开场问题**：开放式，了解大方向
2. **深入问题**：选择题，确认具体需求
3. **细节问题**：追问式，确保理解准确
4. **确认问题**：复述式，获得用户确认

#### 避免的提问方式

- ❌ 一次问太多问题
- ❌ 问太抽象的问题
- ❌ 不给选项只问开放式问题
- ❌ 不根据回答调整问题

## 用户场景分析

### 分析维度

1. **使用频率**
   - 每天：需要高效、稳定
   - 每周：需要易用、可定制
   - 每月：需要简单、快速
   - 偶尔：需要直观、有引导

2. **skill 水平**
   - 新手：需要详细引导、示例
   - 中级：需要灵活配置、高级功能
   - 高级：需要最大控制、可扩展

3. **偏好**
   - 简单：最少功能，快速上手
   - 功能全：完整功能，满足所有需求
   - 创新：新技术栈，前沿方案

4. **场景**
   - 个人：个人偏好，快速迭代
   - 团队：团队规范，协作友好
   - 企业：企业标准，安全合规

## 需求确认

### 确认内容

1. **功能需求清单**
   - 列出所有功能需求
   - 标注优先级（必须/应该/可以）
   - 标注复杂度（简单/中等/复杂）

2. **非功能需求**
   - 性能：响应时间、并发量
   - 安全：数据安全、权限控制
   - 易用性：学习成本、操作步骤

3. **约束条件**
   - 时间：截止日期
   - 预算：资源限制
   - 技术栈：技术限制

### 打分标准

| 维度 | 满分 | 检查项 |
|------|------|--------|
| 需求完整性 | 30 | 功能需求是否完整、非功能需求是否明确 |
| 需求清晰度 | 30 | 需求描述是否清晰、无歧义 |
| 需求可行性 | 20 | 需求是否可行、是否在技术范围内 |
| 需求一致性 | 20 | 需求之间是否一致、无矛盾 |
| **总分** | **100** | |

**通过标准：** ≥ 80 分
