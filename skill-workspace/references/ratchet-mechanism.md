# 棘轮机制 (Ratchet Mechanism)

## 1. 概述

**核心理念：分数只能上升，每一轮要么改进 Skill，要么干净地回滚。**

棘轮机制是 Skill 优化迭代的核心保障。它确保每次优化周期都不会让 Skill 质量倒退——就像机械棘轮一样，只允许朝一个方向转动。如果某轮优化未能带来可量化的提升，系统将自动回滚到上一个已验证状态，保证交付质量单调递增。

---

## 2. 核心规则

### 2.1 只保留改进

- 每轮优化完成后，必须通过量化评估（评分 ≥ 上一轮分数）
- 只有评估通过的版本才能进入下一轮
- 不接受"横向移动"——如果分数持平，视为未改进，触发回滚

### 2.2 自动回滚退步

- 若本轮评分 < 上一轮评分，立即触发自动回滚
- 回滚目标：最近一个通过评估的版本
- 回滚后，本轮的改动被丢弃，不进入后续流程

### 2.3 可追溯链

- 每轮优化必须保留完整的变更记录
- 记录内容包括：
  - 变更前的 SKILL.md 快照
  - 本轮的诊断结果和改进方案
  - 变更后的 SKILL.md 快照
  - 评估分数和通过/回滚决策
- 形成完整的 `v1 → v2 → v3 → ...` 版本链

---

## 3. 实现流程

优化循环遵循 5 个步骤，形成闭环：

```
┌─────────────────────────────────────────────────────────┐
│                   棘轮优化循环                            │
│                                                         │
│   ┌───────────┐    ┌───────────┐    ┌───────────┐      │
│   │  1. 诊断   │───→│  2. 改进   │───→│  3. 执行   │      │
│   └───────────┘    └───────────┘    └───────────┘      │
│         ↑                                  │            │
│         │                                  ↓            │
│   ┌───────────┐                   ┌───────────┐        │
│   │  5. 决策   │←──────────────────│  4. 评估   │        │
│   └───────────┘                   └───────────┘        │
│         │                                                │
│         ↓                                                │
│   [通过] → 进入下一轮  |  [退步] → 回滚到上一版本         │
└─────────────────────────────────────────────────────────┘
```

### 步骤 1：诊断 (Diagnose)

- 分析当前 SKILL.md 的结构完整性
- 检查各章节是否清晰、完整、无歧义
- 识别可量化的问题点（缺失章节、模糊描述、冗余内容等）
- 输出：诊断报告，列出待改进项及优先级

### 步骤 2：改进 (Improve)

- 基于诊断报告，制定针对性改进方案
- 优先处理高优先级问题
- 改进应具有原子性——每次改一个明确的点
- 输出：改进计划（变更 diff）

### 步骤 3：执行 (Execute)

- 将改进计划应用到 SKILL.md
- 保留变更前的快照用于回滚
- 输出：新版 SKILL.md

### 步骤 4：评估 (Evaluate)

- 对新版 SKILL.md 进行全面评分
- 评分维度参考评分模板（结构完整性、描述清晰度、触发条件等）
- 计算本轮分数，与上一轮分数对比，得出 Δ（增量）
- 输出：评估报告（分数、Δ、通过/不通过）

### 步骤 5：决策 (Decide)

| 条件         | 决策   | 动作                              |
|-------------|--------|----------------------------------|
| 本轮分 > 上轮分 | ✅ 通过 | 保留本轮版本，进入下一轮优化          |
| 本轮分 = 上轮分 | ❌ 回滚 | 回滚到上一版本，本轮改动丢弃          |
| 本轮分 < 上轮分 | ❌ 回滚 | 回滚到上一版本，本轮改动丢弃          |

---

## 4. 触顶信号

**当优化收益递减至临界点时，自动停止迭代。**

### 触顶判定规则

```
连续 2 轮 Δ < 2 分 → 自动停止优化
```

具体说明：

- **Δ** = 本轮分数 - 上轮分数
- 当连续 2 轮的增量 Δ 均小于 2 分时，判定为"已触顶"
- 触顶后不再进行新一轮优化，当前版本即为最终版本

### 触顶信号的意义

- 避免无效迭代：分数提升极小时，继续优化的投入产出比过低
- 保护已有成果：在接近最优时，激进改动反而可能引入退步
- 节省资源：及时停止，将注意力转向其他待优化的 Skill

---

## 5. 文件大小约束

**优化后的 SKILL.md 不应超过原始大小的 150%。**

### 约束规则

```
优化后大小 ≤ 原始大小 × 1.5
```

### 约束的目的

- **防止膨胀**：避免通过堆砌内容来提升分数
- **保持精炼**：鼓励用更少的字表达更清晰的含义
- **提高可维护性**：过长的 SKILL.md 难以快速理解和执行

### 超限处理

- 若某轮优化后文件超过 150% 限制，该轮视为"不通过"
- 需要在下一轮中精简内容，将文件大小压回限制范围内
- 精简时优先删除：
  1. 冗余描述
  2. 过于详细的示例（保留关键示例即可）
  3. 重复说明同一概念的内容
