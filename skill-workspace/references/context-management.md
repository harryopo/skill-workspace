# 上下文管理最佳实践

## 1. 概述

### 目标
- **降低幻觉**：通过结构化的上下文管理，减少 AI 对历史信息的错误记忆或虚构
- **提高执行准确性**：确保每一步执行都基于准确的需求理解和已完成的工作成果
- **保持连贯性**：在长对话中维持任务的连续性，避免遗漏关键步骤
- **优化资源使用**：在有限的上下文窗口内，最大化有效信息密度

### 核心原则
- **主动规划优先**：任何任务开始前，先生成明确的执行计划
- **阶段性记忆**：将长任务分解为阶段，每个阶段独立保存关键信息
- **智能压缩**：当上下文过长时，自动压缩而非丢失信息
- **可追溯性**：任何决策和完成状态都应有据可查

## 2. 内置 Plan 机制

### 2.1 开始前：生成整体计划

在执行任何复杂任务之前，必须生成结构化的计划文档：

```
## 任务计划

### 任务目标
[清晰描述最终交付物]

### 任务拆解
1. [阶段1] - [具体子任务] - [预计时间]
2. [阶段2] - [具体子任务] - [预计时间]
...

### 依赖分析
- [任务A] 依赖 [任务B] 的输出
- [任务C] 可并行执行

### 风险识别
- [风险1]：[应对措施]
- [风险2]：[应对措施]

### 成功标准
- [ ] [可验证的完成条件1]
- [ ] [可验证的完成条件2]
```

### 2.2 执行中：分阶段记忆

每个阶段完成时，自动保存该阶段的关键信息到阶段记忆：

```
## 阶段记忆：[阶段名称]

### 完成状态
- 状态：[已完成/进行中/阻塞]
- 完成时间：[时间戳]

### 关键输出
- [输出文件/成果]
- [生成的代码/配置]

### 关键决策
- 决策1：[选择A而非B，原因：...]
- 决策2：[...]

### 下阶段输入
- [需要传递给下一阶段的信息]
```

### 2.3 上下文过长时：自动压缩

当检测到上下文接近限制时，触发压缩流程：

**触发条件判断**：
- 对话轮次 > 20 轮
- 估算 token 数 > 80% 上下文窗口
- 用户明确要求"继续"或"下一步"时检查

**压缩策略**：
1. 保留最近 5 轮完整对话
2. 保留所有阶段记忆摘要
3. 保留当前任务计划
4. 压缩更早的对话为关键点列表

### 3.2 记忆保存时机

| 事件 | 保存内容 |
|------|----------|
| 用户确认需求 | 需求澄清记忆 |

| 功能完成 | 更新开发实现记忆 |
| 测试通过 | 测试部署记忆 |
| 任务完成 | 完整项目总结 |

## 4. 上下文压缩策略

### 4.1 触发条件

**自动触发**：
- 对话轮次 > 20 轮
- 估算上下文使用率 > 80%
- 连续执行超过 30 分钟

**手动触发**：
- 用户说"压缩上下文"、"上下文太长了"
- 用户说"继续"、"下一步"（检查是否需要压缩）

### 4.2 压缩方式

**压缩公式**：
```
压缩后上下文 = 
  最近 5 轮完整对话 +
  所有阶段记忆摘要（每个阶段 ≤ 200 字）+
  当前任务计划 +
  待完成任务列表 +
  关键决策索引
```

**压缩执行步骤**：
1. 识别所有已完成的阶段，生成该阶段摘要
2. 识别未完成阶段的当前状态
3. 保留最近 5 轮对话的完整内容
4. 生成压缩后验证清单

### 4.3 压缩后验证

**必验项目**：
- [ ] 用户原始需求是否保留
- [ ] 当前任务进度是否准确
- [ ] 关键决策和原因是否可追溯
- [ ] 待完成任务是否完整列出
- [ ] 文件路径和代码位置是否保留

**验证失败处理**：
- 如果关键信息丢失，向用户确认后恢复
- 生成"压缩报告"，列出被压缩的内容供用户查阅

### 4.4 压缩报告模板

```markdown
## 上下文压缩报告

### 压缩时间
[时间戳]

### 压缩前状态
- 对话轮次：[N] 轮
- 估算 Token：[N]

### 保留内容
- 最近对话：[N] 轮
- 阶段记忆：[N] 个阶段

### 压缩摘要
[被压缩内容的关键点列表]

### 可恢复信息
- 用户可通过"展开XX阶段的细节"获取更多信息
```

## 5. Plan 生成流程

### 5.1 流程概述

```
用户需求 → 分析 → 拆解 → 依赖识别 → 时间估算 → Plan 文档
     ↑                                            ↓
     └──────── 用户确认/修改 ←────────────────────┘
```

### 5.2 详细步骤

#### 步骤 1：分析用户需求
```
输入：用户的自然语言描述
处理：
  - 提取核心目标
  - 识别约束条件
  - 明确交付物
输出：结构化需求描述
```

#### 步骤 2：拆解任务
```
输入：结构化需求
处理：
  - 识别主要阶段
  - 每个阶段拆解为可执行的子任务
  - 确保每个子任务有明确的完成标准
输出：任务树/WBS
```

#### 步骤 3：识别依赖
```
输入：任务树
处理：
  - 分析任务间的前置关系
  - 识别可并行执行的任务
  - 标记关键路径
输出：依赖图 + 执行顺序建议
```

#### 步骤 4：估算时间
```
输入：任务树 + 依赖图
处理：
  - 为每个子任务估算时间
  - 计算各阶段时间
  - 考虑风险缓冲（建议 +20%）
输出：时间表
```

#### 步骤 5：生成 Plan 文档
```
输入：以上所有分析结果
处理：
  - 整合为结构化 Plan 文档
  - 添加执行检查清单
  - 添加成功标准
输出：Plan.md 或 Plan 内容
```

### 5.3 Plan 文档模板

```markdown
# 任务执行计划

## 基本信息
- 任务名称：[名称]
- 创建时间：[时间]
- 预计总时长：[时间]
- 当前状态：[未开始/进行中/已完成]

## 任务目标
[清晰、可验证的目标描述]

## 任务分解

### 阶段 1：[阶段名称] [预计时间]
- [ ] 子任务 1.1：[描述] [预计时间]
- [ ] 子任务 1.2：[描述] [预计时间]
- 里程碑：[阶段完成标准]

### 阶段 2：[阶段名称] [预计时间]
- [ ] 子任务 2.1：[描述] [预计时间]
- [ ] 子任务 2.2：[描述] [预计时间]
- 里程碑：[阶段完成标准]

## 依赖关系
- 阶段 2 依赖阶段 1 完成
- 子任务 2.1 和 2.2 可并行

## 风险与应对
| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| [风险1] | 中 | 高 | [措施] |

## 成功标准
- [ ] [标准1]
- [ ] [标准2]
- [ ] 所有测试通过
- [ ] 用户验收确认

## 进度追踪
| 阶段 | 计划完成 | 实际完成 | 状态 |
|------|----------|----------|------|
| 阶段1 | [日期] | - | 未开始 |
| 阶段2 | [日期] | - | 未开始 |
```

### 5.4 Plan 更新机制

**何时更新**：
- 阶段完成时：更新状态和实际完成时间
- 需求变更时：重新评估影响范围
- 遇到阻塞时：更新风险和应对措施
- 用户确认新信息时：更新相关部分

**如何更新**：
```
1. 读取当前 Plan
2. 标记变更项
3. 重新计算受影响的时间估算
4. 保存更新后的 Plan
5. 向用户报告变更摘要
```
