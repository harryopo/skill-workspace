# Skill 开发反模式清单

## 概述

本清单整理自 darwin-skill 实战经验和 SkillLens 论文中发现的常见反模式。这些反模式会导致 Skill 质量下降、优化循环失效、或引入难以发现的回归问题。

**核心原则：**
- 发现反模式 → 立即停止当前操作 → 按正确方式重做
- 反模式不是"风格偏好"，是会直接损害 Skill 质量的行为
- 每轮 Phase 2 改动前必须对照本清单检查

## 反例清单

### AP-01: 同 context 自评自改

**错误做法：** 在同一个 context 中既执行修改又执行评分，导致评分结果受修改过程影响，失去客观性。

**正确做法：** 修改完成后，新开 context 进行评分，确保评分环境干净。

**危害：**
- 评分结果不可信，无法真实反映改动效果
- 可能掩盖引入的问题
- 形成"自欺欺人"的闭环

**示例：**
```
❌ 错误：修改 SKILL.md → 同时运行 evals → 根据结果继续改
✅ 正确：修改 SKILL.md → 保存 → 新 context → 运行 evals → 分析结果 → 再决定是否改
```

---

### AP-02: git reset --hard 当回滚

**错误做法：** 使用 `git reset --hard` 回滚改动，丢失工作区所有未提交修改。

**正确做法：** 使用 `git stash` 或 `git checkout -- <file>` 精确回滚特定文件。

**危害：**
- 丢失其他有价值的改动
- 无法恢复被误删的内容
- 破坏工作区状态

**示例：**
```bash
❌ 错误：git reset --hard HEAD
✅ 正确：
   # 回滚单个文件
   git checkout -- SKILL.md
   # 或暂存所有改动
   git stash
   # 或创建临时分支保存
   git checkout -b backup-branch
   git checkout main
```

---

### AP-03: 为凑分增冗余

**错误做法：** 为了提高评分而添加无实际价值的冗余内容（如重复的触发词、无意义的示例、过度的格式化）。

**正确做法：** 评分应反映真实质量，不追求虚高分数。

**危害：**
- 增加 token 消耗
- 降低 Skill 可读性
- 掩盖真正的质量问题
- 可能导致误触发

**示例：**
```
❌ 错误：
   ## 触发条件
   - 用户说"优化"
   - 用户说"改进"
   - 用户说"改善"
   - 用户说"提升"
   - 用户说"加强"  （语义重复，只为凑触发词数量）

✅ 正确：
   ## 触发条件
   - 用户要求优化/改进 Skill
   - 测评发现问题需要修复
   （精简但覆盖完整）
```

---

### AP-04: 跳过 test-prompts 直接评分

**错误做法：** 不运行 test-prompts 就直接进行评分，导致评分缺乏实际测试数据支撑。

**正确做法：** 必须先运行 test-prompts 获取实际输出，再基于输出进行评分。

**危害：**
- 评分脱离实际，可能高估 Skill 质量
- 无法发现实际执行中的问题
- 浪费后续优化时间

**流程：**
```
✅ 正确流程：
   1. 准备 test-prompts
   2. 运行 test-prompts，收集输出
   3. 基于输出进行评分
   4. 分析评分结果
   5. 制定优化方案
```

---

### AP-05: 轮内改多个维度

**错误做法：** 在单轮优化中同时修改多个维度（如同时改触发词、改流程、改输出格式）。

**正确做法：** 每轮只改一个维度，改完验证效果后再决定下一步。

**危害：**
- 无法确定哪个改动产生了效果
- 可能引入交叉影响
- 难以回滚特定改动
- 增加调试复杂度

**示例：**
```
❌ 错误：同时修改
   - description 触发词
   - workflow 步骤顺序
   - 输出格式模板

✅ 正确：
   第 1 轮：只改 description → 验证触发效果
   第 2 轮：只改 workflow → 验证流程效果
   第 3 轮：只改输出格式 → 验证输出效果
```

---

### AP-06: dry_run 比例 > 30%

**错误做法：** dry_run 测试占比超过 30%，导致实际执行测试不足，无法充分验证 Skill 行为。

**正确做法：** dry_run 用于快速验证语法和结构，占比控制在 30% 以内，主要依赖实际执行测试。

**危害：**
- 无法发现运行时问题
- 可能遗漏边界情况
- 过度依赖模拟，脱离实际

**建议比例：**
```
✅ 推荐：
   - dry_run: 20-30% （快速验证）
   - 实际执行: 70-80% （深度验证）
```

---

### AP-07: 静默跳过异常

**错误做法：** 遇到异常时静默跳过，不记录、不报告、不处理。

**正确做法：** 所有异常必须记录日志，根据严重程度决定是否中断流程。

**危害：**
- 隐藏潜在问题
- 导致数据不一致
- 难以定位后续问题的根源

**示例：**
```python
❌ 错误：
try:
    result = risky_operation()
except Exception:
    pass  # 静默跳过

✅ 正确：
try:
    result = risky_operation()
except Exception as e:
    logger.error(f"操作失败: {e}")
    # 根据严重程度决定
    if is_critical(e):
        raise
    else:
        logger.warning("非关键错误，继续执行")
        result = default_value
```

---

### AP-08: 忽视维度相关性单独优化

**错误做法：** 将各个优化维度（触发、流程、输出等）视为完全独立，单独优化每个维度。

**正确做法：** 认识到维度之间存在相关性，优化一个维度时考虑对其他维度的影响。

**危害：**
- 优化效果相互抵消
- 可能引入新的问题
- 无法达到整体最优

**维度相关性示例：**
```
维度关联：
   - 触发词改动 → 影响误触发率
   - 流程改动 → 影响输出格式
   - 输出格式改动 → 影响 token 消耗
   - token 消耗 → 影响响应速度

✅ 正确做法：
   1. 识别维度关联
   2. 优化时评估连锁影响
   3. 必要时联合优化相关维度
```

## 检查时机

**每轮 Phase 2 改动前必须对照本清单检查：**

```
Phase 2 改动前检查清单：

□ AP-01: 是否在新 context 中评分？
□ AP-02: 回滚方式是否安全？（不用 reset --hard）
□ AP-03: 改动是否为真实质量提升？（不为凑分）
□ AP-04: 是否已运行 test-prompts？
□ AP-05: 本轮是否只改一个维度？
□ AP-06: dry_run 比例是否 ≤ 30%？
□ AP-07: 异常是否已记录和处理？
□ AP-08: 是否考虑了维度相关性？

全部通过 → 继续改动
任一未通过 → 先解决问题
```

## 反模式速查表

| 编号 | 反模式 | 关键词 | 一句话纠正 |
|------|--------|--------|------------|
| AP-01 | 同 context 自评自改 | context 混用 | 修改和评分分开 context |
| AP-02 | git reset --hard 当回滚 | 危险回滚 | 用 stash 或 checkout |
| AP-03 | 为凑分增冗余 | 虚假繁荣 | 只加有价值的内容 |
| AP-04 | 跳过 test-prompts 直接评分 | 无测试评分 | 先跑测试再评分 |
| AP-05 | 轮内改多个维度 | 多维同改 | 每轮只改一个维度 |
| AP-06 | dry_run 比例 > 30% | 模拟过多 | 主要靠实际执行 |
| AP-07 | 静默跳过异常 | 异常吞没 | 所有异常必须记录 |
| AP-08 | 忽视维度相关性单独优化 | 孤立优化 | 考虑维度间影响 |
