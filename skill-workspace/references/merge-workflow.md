# Skill 合并工作流

## 概述

合并 skill 是将多个相关 skill 整合为一个更强大、更完整 skill 的过程。

### 合并目标

1. **消除冗余** — 减少功能重叠，降低维护成本
2. **增强能力** — 集成多个 skill 的优点，形成更强功能
3. **优化体验** — 统一触发条件和输出规范，减少用户困惑
4. **解决问题** — 修复原 skill 的痛点和缺陷

### 合并意义

- **对用户** — 一个 skill 解决多种需求，减少选择成本
- **对维护** — 统一维护入口，降低维护复杂度
- **对质量** — 集成最佳实践，提升整体质量

---

## 合并流程

### 第一步：审查原 skill

**目标** — 了解原 skill 的质量现状

**操作** — 调用 review 子技能，对每个原 skill 进行 10 维度打分：

1. **触发准确度** — 触发词是否精准
2. **流程完整度** — 工作流是否完整
3. **输出质量** — 输出是否符合规范
4. **安全性** — 是否存在安全风险
5. **可维护性** — 代码是否易于维护
6. **文档质量** — 说明是否清晰
7. **测试覆盖** — evals 是否充分
8. **Token 效率** — 是否浪费 token
9. **用户体验** — 使用是否顺畅
10. **通用性** — 适用范围是否合理

**输出** — 每个 skill 的审查报告（评分 + 优缺点）

---

### 第二步：理解原 skill

**目标** — 深入理解每个 skill 的设计意图

**操作** — 阅读每个 skill 的 SKILL.md，重点关注：

- **触发条件** — 什么时候被激活
- **工作流程** — 具体执行步骤
- **输出规范** — 期望的输出格式
- **子技能** — 依赖的子技能
- **Gotchas** — 已知的坑和注意事项
- **references** — 参考资料

**输出** — 每个 skill 的理解摘要

---

### 第三步：询问用户

**目标** — 明确合并方向和用户需求

**询问内容**：

1. **保留优点** — 哪些功能/特性必须保留？
2. **解决痛点** — 哪些问题必须解决？
3. **新需求** — 有没有需要新增的功能？
4. **优先级** — 哪些功能最重要？
5. **约束条件** — 有没有特殊限制？

**输出** — 用户需求清单（优先级排序）

---

### 第四步：网上调研

**目标** — 搜索最佳实践和解决方案

**搜索内容**：

1. **解决方案** — 针对痛点的现有方案
2. **最佳实践** — 行业推荐做法
3. **踩坑经验** — 他人遇到的问题
4. **开源项目** — 可参考的实现

**搜索策略**：

- 使用 multi-search-engine 进行多引擎搜索
- 搜索 GitHub 上的类似项目
- 搜索技术博客和论坛

**输出** — 调研报告（方案 + 经验 + 参考）

---

### 第五步：综合优化

**目标** — 设计合并方案

**优化原则**：

1. **保留优点** — 原 skill 的优秀设计必须保留
2. **解决痛点** — 原 skill 的问题必须修复
3. **集成新功能** — 用户需求和调研结果要融入
4. **简化结构** — 减少不必要的复杂度
5. **统一规范** — 触发条件和输出格式统一

**输出** — 合并方案文档（详见输出格式）

---

### 第六步：正常工作流

**目标** — 按标准流程开发合并后的 skill

**流程**：

1. **分阶段开发** — 按方案逐步实现
2. **循环审查** — 每阶段完成后审查
3. **用户确认** — 每阶段完成后确认
4. **本地交付** — 最终交付完整 skill

**输出** — 合并完成的 skill

## 用户确认流程

### 确认时机

每个阶段完成后都需要用户确认：

1. **审查完成后** — 确认审查结果是否准确
2. **理解完成后** — 确认理解是否正确
3. **需求收集后** — 确认需求是否完整
4. **调研完成后** — 确认调研结果是否可用
5. **方案设计后** — 确认合并方案是否满意
6. **开发完成后** — 确认最终成果是否符合预期

### 确认方式

```markdown
## ✅ 阶段确认

**当前阶段：** [阶段名称]

**已完成：**
- [已完成内容 1]
- [已完成内容 2]

**产出：**
- [产出物 1]
- [产出物 2]

**请确认：**
1. [ ] 审查结果准确
2. [ ] 产出物符合预期
3. [ ] 可以进入下一阶段

**反馈：**
[用户填写反馈]
```

### 处理反馈

- **确认通过** — 进入下一阶段
- **需要修改** — 根据反馈修改后重新确认
- **方向调整** — 回到第三步重新收集需求

## Gotchas

### G1: 合并不是简单拼接
- 合并不是把多个 skill 的内容复制到一起
- 需要重新设计工作流程，消除冗余

### G2: 保留优点比解决缺点更重要
- 用户已经习惯的优点不能丢
- 新方案必须兼容原有使用习惯

### G3: 用户确认是必要环节
- 每个阶段都要用户确认
- 不要假设用户的意图

### G4: 调研要充分
- 不要闭门造车
- 搜索最佳实践和踩坑经验

### G5: 测试要覆盖
- 合并后的 skill 必须有完整的 evals
- 确保功能不退化
