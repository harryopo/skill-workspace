# Skill 审查评分标准

## 审查流程

```
安全审查 → 定位 Skill → 10 维度评分 → 生成报告
```

**安全审查是前置条件：** 发现安全问题（BLOCK）则停止，不进入质量评分。

---

## 安全审查标准

### 安全评级

| 评级 | 条件 | 后续动作 |
|------|------|----------|
| ✅ SAFE | 无风险标记 | 继续质量评分 |
| ⚠️ WARNING | 有低/中风险 | 继续评分，标注风险 |
| 🔴 DANGER | 有高风险 | 继续评分，重点标注 |
| 🛑 BLOCK | 有 Critical 风险 | 停止，不进入评分 |

### Critical 风险（直接 BLOCK）

- 引用 `~/.ssh`、`~/.aws`、`~/.env` 等敏感路径
- 使用 `curl`、`wget`、`nc`、`bash -i` 等命令
- `base64` 混淆内容
- 禁用安全机制
- 未知或可疑 URL
- `network` + `shell` 同时出现

### Warning 风险

- `/**/*` 等宽泛通配符
- `sudo` 使用
- 潜在的提示注入

## A 组：规范性审查

### A1. 触发（Trigger）— 20 分

**检查项：**
- [ ] description 前 250 字符包含触发词
- [ ] 触发词覆盖用户常用说法（至少 3 个）
- [ ] 有明确的不触发条件
- [ ] 不会误触发其他 Skill

**评分：**
| 分数段 | 标准 |
|--------|------|
| 18-20 | 触发词精准，边界清晰，不会误触发 |
| 14-17 | 触发词基本覆盖，偶有模糊 |
| 10-13 | 触发词不足，容易误触发 |
| 0-9 | 无触发词或严重误触发 |

### A2. 结构（Structure）— 15 分

**检查项：**
- [ ] 遵循目录结构（SKILL.md + references/ + assets/ + scripts/）
- [ ] 正文 ≤500 行
- [ ] 参考资料放 references/ 目录
- [ ] 使用 ${SKILL_DIR} 引用文件

**评分：**
| 分数段 | 标准 |
|--------|------|
| 13-15 | 结构完整，符合规范 |
| 10-12 | 结构基本完整，有小缺漏 |
| 7-9 | 结构不完整，缺少关键目录 |
| 0-6 | 无目录结构，内容堆砌 |

### A3. 上下文（Context）— 15 分

**检查项：**
- [ ] 复杂任务使用 context: fork
- [ ] 需要对话历史时使用 inline
- [ ] 三层加载合理（元数据/正文/资源）
- [ ] 上下文消耗可控

**评分：**
| 分数段 | 标准 |
|--------|------|
| 13-15 | 上下文策略正确，消耗可控 |
| 10-12 | 基本正确，有优化空间 |
| 7-9 | 选择不当，上下文浪费 |
| 0-6 | 严重错误，上下文爆炸 |

### A4. 安全性（Security）— 10 分

**检查项：**
- [ ] 无敏感信息泄露（API key、密码）
- [ ] 无命令注入风险
- [ ] allowed-tools 最小权限
- [ ] 危险操作有警告

**评分：**
| 分数段 | 标准 |
|--------|------|
| 9-10 | 安全无风险 |
| 7-8 | 有小风险，需注意 |
| 4-6 | 有明显风险 |
| 0-3 | 严重安全问题 |

### A5. 可维护性（Maintainability）— 10 分

**检查项：**
- [ ] 内容模块化
- [ ] 有版本管理建议
- [ ] 有退役/合并条件说明
- [ ] 个人 vs 团队使用明确

**评分：**
| 分数段 | 标准 |
|--------|------|
| 9-10 | 高度可维护 |
| 7-8 | 基本可维护 |
| 4-6 | 维护困难 |
| 0-3 | 难以维护 |

### A6. 测试（Testing）— 10 分

**检查项：**
- [ ] 有 evals 评测集
- [ ] 有边界样例和反例
- [ ] 有验收清单
- [ ] 有弱模型验收建议

**评分：**
| 分数段 | 标准 |
|--------|------|
| 9-10 | 测试完整 |
| 7-8 | 有基本测试 |
| 4-6 | 测试不足 |
| 0-3 | 无测试 |
