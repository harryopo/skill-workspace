# Skill 开发实战方法论

> 来源：社区实战经验总结
> 补充官方规范之外的开发、验证和迭代方法

---

## 一、判断任务是否值得 Skill 化

### 先跑 baseline，再决定

不要急着写 Skill。先拿 3-5 个真实 case 让 Agent 在没有 Skill 的情况下完整跑一遍。观察它是否稳定暴露同一类问题：

- 总是漏掉某个检查步骤
- 总是误用同一个工具
- 总是输出无法验收
- 每次都需要你重新解释同一套口径

**如果问题不稳定（有时对有时错），说明场景边界还不够清晰，不适合 Skill 化。**

### 适合第一批的场景

| 场景 | 为什么适合 |
|------|-----------|
| 需求澄清 | 输入模糊，但产出格式可固定 |
| 技术方案生成 | 流程稳定，需要统一模块、接口、取舍说明 |
| 代码 Review | 关注点明确，容易形成检查清单 |
| 会议纪要加工 | 信息来源固定，输出可标准化 |
| 文档整理 | 有固定排版要求和发布规范 |

### 三种常见拆法

1. **按任务阶段拆** — 读取 → 分析 → 输出，每个阶段一个 Skill
2. **按输入类型拆** — 草稿整理、会议纪要、技术方案，各一个 Skill
3. **按输出格式拆** — 知识库文章、PR 描述、周报，各一个 Skill

**可以组合，但不建议都塞进一个 Skill。边界越清楚，后面越好维护。**

## 二、Description 写作进阶

### 三个自检问题

写 description 前先回答：

1. **用户真实会怎么提出这个需求？** — 用他们的话，不是你的抽象
2. **哪些相邻需求不应该触发？** — 明确边界，避免误触发
3. **加载这个 Skill 后，Agent 应该产出什么类型的结果？** — 预期产出要清晰

如果这三个问题答不清，先不要急着写正文。

### 好 description 的结构

```
前 50 字符：核心功能（做什么）
前 100 字符：触发关键词（用户怎么说）
前 200 字符：产出类型（输出什么）
前 250 字符：边界提示（不做什么）
```

### 改 description 要同步补评测

只要改 description，最好同步补两类评测：
- **正例**：必须触发的场景
- **反例**：看起来相似但禁止触发的场景

这样才能判断修改是真的改善了路由，还是只是让 Skill 看起来更完整。

## 三、脚本化稳定动作

### 为什么要脚本化

模型擅长理解意图和生成内容，但不擅长精确记住每个工具的参数组合。如果一个动作每次都一样（读哪个文件、用什么参数、怎么处理失败），把它写成脚本或 CLI 示例，比让模型每次"想起来"稳定得多。

### 适合脚本化的动作

- 每次都执行且参数固定的动作（如读取文档目录、校验 JSON 格式）
- 涉及多步组合的操作（如先 fetch → 判断结构 → 按 section 精读）
- 失败后有固定重试或回退路径的动作
- 需要精确参数、人容易记错的命令

### 脚本化 vs 写进 workflow 的判断标准

```
参数固定 + 每次都执行 → 脚本化
参数变化 + 需要判断 → 写进 workflow
偶尔执行 + 简单命令 → 写在 workflow 里即可
```

### 在 Skill 里写 CLI 示例的正确方式

不要只写"读取文档内容"。要给出命令、参数选择逻辑和失败处理：

```markdown
# 第一步：读取文档目录，判断结构
lark-cli docs +fetch --doc "$DOC_URL" --scope outline

# 第二步：按章节精读（不要全量 fetch）
lark-cli docs +fetch --doc "$DOC_URL" --scope section --start-block-id "$HEADING_ID"

# 权限失败时：提示用户执行授权
# lark-cli auth login --scope "docx:document:readonly"
```

## 四、验收清单

写完第一版 Skill 后，用这个清单检查：

1. ✅ 用户换一种常见说法时，description 仍能触发
2. ✅ 缺少必要输入时，Skill 知道该追问、假设还是停止
3. ✅ 每个步骤都有明确动作，而不只是原则性描述
4. ✅ 输出格式固定，能被人或下一个流程继续使用
5. ✅ 至少有 3 个标准样例、2 个边界样例、1 个反例
6. ✅ 关键 CLI 或脚本调用有参数示例和失败处理
7. ✅ 停止条件明确：缺资料、权限失败、高风险动作、连续失败时有处理路径

## 五、用 Evals 抓回归

### 什么是 Evals

给一组输入，让 Skill 产出结果，再按标准判断结果好不好。它不是为了证明作者写得对，而是为了快速发现回归。

### 最小 eval 文件结构

```json
{
  "skill_name": "my-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "用户的真实任务描述",
      "expected_output": "期望 Agent 最终交付什么",
      "assertions": [
        { "id": "a1", "text": "可检查的细项1" },
        { "id": "a2", "text": "可检查的细项2" }
      ]
    }
  ]
}
```

### 样本建议

| 类型 | 数量 | 用途 |
|------|------|------|
| 核心样本 | 5-10 个 | 用户最常提交的任务 |
| 边界样本 | 3-5 个 | 空输入、资料不全、格式混乱 |
| 已知坑 | 3-5 个 | 之前误触发、乱用工具的案例 |

### 评测策略

- **新建 Skill**：比较"有 Skill vs 无 Skill"
- **优化 Skill**：比较"新版 vs 旧版"
- 每次改 Skill 后用同一把尺子回看，才知道是真变好还是看起来更复杂

## 六、Gotchas 飞轮：把失败变成边界

### 什么是 Gotchas

不要每次遇到问题就重写整份说明。更有效的做法是把真实失败追加成 gotcha：

- 什么情况下不要触发
- 哪个工具容易误用
- 哪类输入看起来相似但其实不该走这条流程
- 连续失败时应该停在哪里

### Gotchas 的价值

它来自真实任务，而不是想象中的完美流程。一个 Skill 越成熟，它不一定越长，但会更清楚地知道哪些路不能走。

### 维护方式

```markdown
## Gotchas

### G1: 看起来像但不该触发
- 用户说"帮我看看这段代码"可能是问功能，不是审查
- 判断依据：是否提到"质量"、"安全"、"性能"等关键词

### G2: 容易误用的工具
- 不要用 Write 直接覆盖用户文件，先用 Edit 做 diff
- 原因：Write 会丢失用户未提交的修改

### G3: 连续失败时停止
- 如果连续 3 次输出不符合格式要求，停下来问用户
- 不要无限重试
```

## 七、从真实协作沉淀 Skill

### 沉淀路径

1. **先在聊天里完成一次真实任务** — 不急着抽象，让 Agent 暴露理解偏差和步骤遗漏
2. **人持续纠偏** — 哪些地方必须确认、哪些动作不能自动做、哪些输出不算合格，都要在对话中说清楚
3. **让强模型复盘整段聊天** — 提炼触发条件、输入要求、固定步骤、失败处理和验收标准
4. **沉淀 SKILL.md** — 长资料放 references/，稳定动作做成 scripts/ 或 CLI 示例
5. **换一个同类任务验证** — 让较弱模型按 Skill 执行，看它是否还能完成
6. **根据失败点继续改** — 补触发边界、补反例、补检查清单

### 复盘时重点看什么

- 哪些判断是每次都要做的 → 写进 workflow
- 哪些判断是场景相关的 → 写进 gotchas 或 examples
- 哪些资料需要反复引用 → 放进 references/
- 哪些动作参数固定 → 做成 scripts/

### 强模型写，弱模型跑

复杂、模糊、需要归纳经验的部分，交给更强的模型来整理。一旦流程被写成 Skill，后续同类任务就可以由更便宜、更快的模型按步骤执行。

**验收标准：如果弱模型也能按流程产出稳定结果，说明 Skill 真的把经验写进了流程。**

## 八、泛化与特化的时机

### 从特化开始

建议从特化开始，用真实 case 验证。当同一类规则在多个 case 里反复出现，再把它泛化。

### 什么时候泛化

- 同一条规则在 3+ 个不同 case 里重复出现
- 用户主动要求"这个也适用另一个场景"
- 边界条件已经验证清楚

### 什么时候特化

- 泛化后误触发率上升
- 不同场景的输出格式差异太大
- 用户反馈"这个 Skill 什么都能做但什么都做不好"

---

## 九、什么时候该退役

不是所有 Skill 都应该永远存在。以下情况可以考虑合并、降级或删除：

- 模型原生能力已经覆盖该场景
- 维护成本超过收益
- 流程已经过时
- 和新 Skill 大量重叠

**退役不是失败。** Skill 太多而且互相打架，Agent 反而更难做对。留下能被复用、能被测试、能被别人接手维护的那部分就好。

---

## 十、迭代循环

```
遇到 case → 跑 Skill → 记录问题 → 改 description / workflow / examples / scripts → 用下一个 case 验证
```

每次失败都是改进的素材。不要把失败当作问题，把它当作下一版规则、样例和评测的来源。
